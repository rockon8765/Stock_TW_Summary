# 壽險股儀表板客戶簡報 — Slide 大綱

## Context（背景）

`feature/layout-claude` 分支的 14 區塊個股儀表板已完成並推上 GitHub。使用者（你）要向基金經理人 / 投資委員會做簡報，說明這個儀表板的設計理念。

**決定的簡報參數：**
- 格式：**Slide 大綱**（投影片標題 + 每張要點 + 口語稿）
- 聽眾：**基金經理人 / 投資委員會**（中技術：重決策感、不愛公式細節）
- 主軸：**「專為壽險所設計」的 domain-fit**（所有新區塊都對齊壽險決策需求）

目標：15 分鐘可講完，聽眾能答得出「這個報表幫我解決了什麼？」與「它為什麼長這樣？」

---

## 整體敘事弧線

> **痛點 → 壽險決策特性 → 結構如何對應 → 實測驗證 → 路線圖**

不講技術細節，講「為什麼這樣設計對你有用」。

### Slide 5-bis：關鍵名詞速查 — FCF 股利覆蓋（speaker notes）

當有人問「什麼是 FCF 股利覆蓋？」時的現場應答稿：

**一句話**：
> 公司的自由現金流，夠不夠支撐它發出去的現金股利。

**公式**：
```
FCF 股利覆蓋 = TTM 自由現金流量 ÷ TTM 現金股利發放
```
（TTM = 最近 4 季滾動加總）

**為什麼壽險特別在意**：
- 壽險買股票主要不是賺價差、是賺穩定現金股利（對應保單未來給付）
- 只看「**股利發放率 = 股利 / EPS**」會被假象騙：**EPS 是帳面數字（含折舊等非現金項）**，**股利是真的從銀行戶頭匯出去的現金**
- 公司可以帳面 EPS 漂亮、實際現金不夠，只好**借錢發股利**。景氣反轉 → 突然減配 → 股價崩 → 壽險 portfolio 挨打
- 這個比率**直接揭露此類陷阱**

**三種情境**：

| 比值 | 意義 | 判讀 |
| ---: | --- | --- |
| > 1x | FCF > 股利 | ✅ 健康：用真正賺的錢發股利 |
| ≈ 1x | 剛好打平 | ⚠️ 臨界：無緩衝、景氣反轉可能被迫減配 |
| < 1x | FCF < 股利 | ❌ 警示：**靠借錢、賣資產或動用現金在發股利** |

**2330 實例**：
- 最近 4 季自由現金流 ≈ 7,133 億
- 最近 4 季現金股利發放 ≈ 4,668 億
- 覆蓋倍數 = **1.53x** → 每發 1 元股利，背後有 1.53 元 FCF 撐著 → 健康

**警示門檻**：
- < 1x 顯示**紅字**
- 銀行業顯示「**—**」（自由現金流概念在銀行業不適用；強算會誤導）

**一句收束**：
> 我們寧可空白也不要產生假訊號 — 這也是整個儀表板貫穿的設計原則。

### Slide 6：設計重點① — 方法論上的嚴謹

**不靠直覺、靠資料**：

- **FCF 用 API 官方欄位、不自己推導**
  OCF + 投資活動現金流 ≠ 自由現金流（IFRS 把購買金融資產算在投資活動）。2330 抽樣曾差到 100M+
- **股利季→年聚合**
  原本「近 10 年」標題下放 10 季資料（≈ 2.5 年）。已修正為真正年度聚合
- **方案 B：未結年度不推估發放率**
  2025 的發放率顯示「—」，而不是用半年 EPS 推全年（寧可空白也不要誤導）
- **欄位名 Unicode 安全**
  `母公司業主–稅後純益` 中是全形 en dash（U+2013），用常數管理避免肉眼錯字

**口語稿**：
> 這張是跟投委會交代「我們踩過哪些坑並修掉」。精神是：寧可留空白也不要產生假訊號。

### Slide 7：設計重點② — 警示色語言

**紅字 / 紅底的觸發條件**（讓肉眼 scan 就能抓重點）：

| 指標 | 警示門檻 |
| --- | --- |
| 負債比 | > 50% → 紅字 |
| 流動比 | < 1x → 紅字 |
| 利息保障倍數 | < 3x → 紅字 |
| FCF 股利覆蓋 | < 1x → 紅字 |
| 設質比 | > 30% 紅字，> 50% **紅底** |
| Beta 250D | > 1.2 → 紅字（高系統性風險）|

**口語稿**：
> 不用看具體數字也能先 scan 紅色。把分析師的「精力焦點」從「讀表」改為「看紅點」，再往下深挖。

### Slide 8：數字說話 — 三檔跨產業驗證

**實測截圖對比**（用 2330 / 2412 / 2886 三檔 screenshot 並列）：

| 指標 | 2330 半導體 | 2412 電信 | 2886 金融 |
| --- | ---: | ---: | ---: |
| ROE (TTM) | 35.39% | 10.04% | 9.26% |
| 負債比 | 31.16% | 25.21% | **92.32%** |
| 流動比 | 2.62x | 1.49x | 48.30x |
| 利息保障 | 103.2x | 136.7x | **—** |
| FCF 股利覆蓋 | 1.53x | **—** | **—** |
| Beta 250D | 1.18 | — | — |

**說明**：
- 半導體：典型**成長型高品質**（ROE 35%、低負債、FCF 覆蓋良好）
- 電信：典型**現金牛**（發放率 95-104%，波動低，資本效率中等）
- 金融：**業別不同、公式不適用**（銀行負債比本來就高，利息費用是營收、不是費用）→ 系統**主動顯示「—」+ graceful degradation**，不硬算假數字

**口語稿**：
> 這不是 bug、是 feature。不同產業本來就該出不同的 pattern。系統知道什麼時候「該說我不知道」，才是對你有用的工具。

**可能的問答方向預備**：
1. Q：為什麼不用 Bloomberg？
   A：Bloomberg 是通用工具；這個是壽險決策特化，搜尋摩擦更低、每個欄位都有設計意圖。
2. Q：資料來源可靠嗎？
   A：Dottdot API 是金管會欄位一致的來源，我們有做 live 抽樣驗證、設質等關鍵欄位都已對照。
3. Q：維護成本？
   A：模組化，新增區塊 1-2 天，且每個模組 <200 行。
4. Q：跟其他券商研究平台的差異？
   A：我們不做「買賣建議」，我們做「讓你自己判斷得更快」的資料整合工具。
